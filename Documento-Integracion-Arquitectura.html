1Resumen ejecutivo

Vista de alto nivel del estado actual, sin supuestos de diseño futuro.

El repositorio C:\Sistema Integrado ya opera como plataforma unificada: una SPA React consume una API FastAPI única y persiste en MongoDB. El sistema implementa aislamiento multi-aplicación por aplicacion_id, autenticación JWT, administración de usuarios/squads y módulos de negocio para liquidación y carga de trabajo.

FRONTEND

React + TypeScript + Vite

Rutas protegidas, selector de aplicación, dashboards y vistas operativas para requerimientos, personas, asignaciones, capacidades, estimaciones y Azure DevOps.

BACKEND

FastAPI + Beanie

Routers por dominio, servicios de ANS/liquidación/sincronización y scheduler con APScheduler. Puede servir también el build del frontend en web/dist.

DATOS

MongoDB unificado

Colecciones de plataforma y operativas, documentos embebidos para requerimientos y asignaciones, e índices por aplicación para garantizar aislamiento y rendimiento.

3Decisiones de arquitectura vigentes

Decisiones confirmadas por implementación.

  • Backend único: toda la API vive bajo /api en FastAPI.
  • Persistencia documental: MongoDB con pymongo.AsyncMongoClient + Beanie (no SQLite).
  • Multi-aplicación: aislamiento por aplicacion_id y cabecera X-Aplicacion.
  • Modo consolidado: centinela __todas__ para roles administrativos en lectura.
  • Frontend único: React Router + contexto global para token y aplicación activa.
Nota de consistencia terminológica En el código, el concepto de unidad funcional/tenant se gestiona como Aplicacion y en UI se presenta como Squad. Este documento usa ambos términos como equivalentes operativos.

5Modelo multi-aplicación

Aislamiento de datos y comportamiento de contexto activo.

Resolución del contexto

  • La cabecera X-Aplicacion define el ámbito de consulta/escritura.
  • contexto_aplicacion valida acceso del usuario y retorna filtros Mongo por aplicación.
  • contexto_escritura bloquea escrituras cuando X-Aplicacion=__todas__.

Ciclo de vida de una aplicación

  1. Se crea en /aplicaciones (solo superadmin).
  2. provisionar_aplicacion() crea estructura base (categorías y parámetros).
  3. No se clonan datos operativos (requerimientos, personas, asignaciones, etc.).
Bootstrap inicial En cada arranque, bootstrap() garantiza aplicación inicial y superadmin; la provisión es idempotente.

6Seguridad, autenticación y autorización

Esquema implementado de control de acceso.

Autenticación

  • POST /auth/login valida credenciales y emite JWT.
  • Contraseñas con bcrypt (hash + verificación).
  • GET /auth/me devuelve identidad actual a partir del token.

10Flujo principal de datos

Secuencia principal de operación entre frontend, backend y MongoDB.

  1. Usuario inicia sesión en /login → POST /auth/login.
  2. Frontend almacena JWT + usuario y consulta aplicaciones visibles en /aplicaciones.
  3. Usuario selecciona squad (o consolidado) y el cliente adjunta X-Aplicacion en cada petición.
  4. Backend valida token y autorización; resuelve contexto con contexto_aplicacion.
  5. Routers consultan/actualizan colecciones filtrando por aplicacion_id.
  6. Servicios de dominio calculan ANS, liquidación y sincronizaciones; eventos críticos se registran en bitácora.
  7. Dashboards consumen endpoints agregados y renderizan KPIs en tiempo real.
Flujos adicionales implementados Estimaciones: carga de Excel en base64, persistencia estructurada y creación opcional de tareas en Azure DevOps.
Sincronización AzDO: descarga de work items por iteración y consolidación en azdo_work_items + azdo_sync_log.

11Integraciones y operación

Componentes transversales relevantes para la operación diaria.

Azure DevOps

  • Configuración en azdo_config con alcance app/squad/usuario.
  • Endpoints para probar conexión, listar proyectos/iteraciones, descubrir campos requeridos y sincronizar.
  • Sincronización manual (/azdo/sync) y automática con scheduler.

Scheduler

app/services/scheduler.py ejecuta revisión cada 30 minutos y dispara sincronizaciones según intervalo configurado (manual, hourly, daily).

Importación de datos

Existe importador de Excel (app/importer/excel_importer.py) para la bitácora general, con carga de requerimientos, entregas y catálogos asociados.

13Riesgos técnicos actuales

Riesgos observados directamente en el código actual.

RiesgoImpactoDetalle observado
Configuración AzDO HITSS/EPM no diferenciada en backendAltoEl frontend envía target (HITSS/EPM), pero el router /azdo no separa esa dimensión; puede mezclar configuración entre paneles.
Ruta de importación no registrada en router principalMedioExiste api/app/api/importacion.py, pero no está incluida en api/app/api/router.py, por lo que el endpoint no queda expuesto.
Dashboard consolidado backend aún parcialMedio/dashboard/consolidado reporta principalmente personas/categorías y mantiene TODO para métricas de negocio extendidas.
Configuración global sin dependencia explícita de autenticaciónMedio/configuracion no aplica usuario_actual ni requiere_rol; requiere endurecimiento según política de seguridad del entorno.

14Próximos pasos recomendados

Acciones priorizadas para cerrar brechas detectadas.

  1. Separar configuración AzDO por destino (HITSS/EPM) en backend y persistencia.
  2. Decidir el estado de importación Excel: exponer /importacion en router principal o retirar código/UI no usados.
  3. Endurecer seguridad de /configuracion con autenticación y control de rol.
  4. Completar métricas de /dashboard/consolidado para alinear backend y dashboards de negocio.
  5. Agregar pruebas automatizadas focalizadas para autenticación, contexto multi-aplicación y modo consolidado.
  6. Actualizar documentación operativa (backup MongoDB, monitoreo de scheduler y fallos de sync AzDO).

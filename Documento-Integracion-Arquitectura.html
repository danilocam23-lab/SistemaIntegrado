1Resumen ejecutivo

Qué se va a integrar, qué exige la nueva versión y cuál es el resultado esperado.

HITSS opera hoy dos aplicaciones independientes que gestionan información parcialmente solapada sobre el mismo flujo de trabajo EPM-HITSS, y existe además un conjunto de documentos de "nueva versión" que exige convertir la plataforma en un producto multi-aplicación.

SISTEMA A

Sistema Liquidador EPM-HITSS

Ciclo Solicitud → Requerimiento → Entregas → Facturación, con ANS, tarifas y liquidación. Python/FastAPI, SQLite, Jinja2 + HTMX.

SISTEMA B

Workload Manager

Carga de trabajo de desarrolladores: asignaciones, sprints, capacidad, Azure DevOps y catálogo de proyectos. Node.js/Express, SQLite, React.

DOCUMENTOS "NUEVA VERSIÓN"

Requisitos multi-aplicación

Exigen una plataforma multi-tenant sobre MongoDB, con autenticación, roles/permisos y un módulo de administración de aplicaciones.

El solapamiento que justifica la integración El project_catalog del Workload Manager es una copia parcial de los Requerimientos del Liquidador: ambos importan del mismo Excel "BITÁCORA GENERAL" y comparten códigos (code/codigo_sc, req_code/codigo_req), aplicativos, estados y el LT EPM. Los developers equivalen a las Persona del Liquidador, y la capacidad mensual existe en ambos.
Lo que añade la "nueva versión": multi-aplicación Los documentos requerimientos-multiaplicacion.md y manual-dummies-migracion-mongo.md (incorporados al proyecto en la carpeta referencias/) elevan el alcance: la plataforma debe soportar varias aplicaciones (CRM, BI, Soporte, Mesa de Ayuda…) sobre una sola base, aisladas por un campo aplicacion_id. Cada aplicación tiene sus propios datos; una aplicación nueva nace con estructura base pero sin datos operativos. Esto exige también autenticación, roles/permisos y un módulo de administración.

Resultado esperado

  • Una única base de datos MongoDB como fuente de verdad.
  • Un único backend Python/FastAPI que expone una API REST.
  • Una única interfaz React (SPA) con todos los módulos, login y selector de aplicación.
  • Multi-aplicación: los datos de hoy se migran como una aplicación inicial; se pueden crear nuevas aplicaciones aisladas desde administración.
  • Seguridad: autenticación JWT, contraseñas con bcrypt y autorización por roles.
  • Vista consolidada: los roles de administración ven todas sus aplicaciones a la vez en un dashboard unificado con gráficas comparativas.
  • El Requerimiento se captura una sola vez y alimenta liquidación y planeación de carga.

3Decisiones de arquitectura

Decisiones confirmadas que enmarcan todo el plan.

① Backend

FastAPI único. El backend Express se porta a routers FastAPI.

② Base de datos

MongoDB local en el servidor IIS, como servicio de Windows.

③ Frontend

React SPA único; FastAPI queda como API REST pura.

④ Multi-aplicación

Plataforma multi-tenant aislada por aplicacion_id.

Implicaciones de adoptar MongoDB

  • Desaparece Alembic. MongoDB no tiene esquema rígido; la evolución del modelo se gestiona en el código.
  • Desaparecen SQLAlchemy y node:sqlite. El acceso a datos se hace con Beanie (ODM asíncrono sobre Pydantic v2) y el driver Motor.
  • La integridad referencial pasa a la aplicación: se embeben documentos (1:1 y 1:pocos) o se referencian por ObjectId; se valida en la capa de servicios.
  • Las consultas agregadas se reescriben como aggregation pipelines.

Implicaciones de adoptar el modelo multi-aplicación

  • Aparecen dos colecciones de plataforma: aplicaciones (los tenants) y usuarios (cuentas de acceso).
  • Toda la data operativa lleva aplicacion_id y toda consulta o inserción debe filtrar por él — es la regla más crítica del diseño.
  • Se necesita un middleware de resolución de aplicación que determine y valide la aplicación activa en cada petición.
  • Crear una aplicación nueva clona la estructura base (categorías, estados, configuración) pero no copia datos de negocio.
Decisión que reemplaza a la versión 1.0 de este documento La v1.0 proponía sesión por cookie. Atendiendo el documento requerimientos-multiaplicacion.md, la autenticación pasa a JWT y el hash de contraseñas a bcrypt. Ver sección 6.

Aclaración de un choque de nombres

"Aplicación" tiene dos significados — se desambiguan así El Liquidador ya tiene una entidad Aplicacion (el software o aplicativo al que apunta una solicitud, ligado a una Dirección). La nueva versión usa "aplicación" para el tenant. Para evitar ambigüedad:
  • aplicaciones → colección de tenants (CRM, BI, Soporte).
  • aplicativos → la antigua Aplicacion del Liquidador (software objetivo del requerimiento). renombrar

5Modelo multi-aplicación

Cómo una sola plataforma y una sola base sirven a varias aplicaciones aisladas.

La plataforma no crea un sistema distinto por aplicación: existe un solo backend, un solo frontend y una sola base MongoDB. La separación es lógica, mediante un campo aplicacion_id presente en todo documento operativo.

Concepto

Una plataforma

Mismo código base para todas las aplicaciones. No se duplica desarrollo.

Datos aislados

Cada documento operativo pertenece a una aplicación. Ninguna consulta cruza el límite de su aplicacion_id.

Aplicación inicial

Todos los datos migrados de Liquidador + Workload Manager quedan bajo una aplicación inicial (p. ej. código epm-hitss).

Ciclo de vida de una aplicación

  1. Un superadmin crea la aplicación desde el módulo de administración (nombre, código, descripción).
  2. El sistema ejecuta el servicio de provisión: crea la configuración base, las categorías base y los estados base de esa aplicación.
  3. La aplicación aparece en el selector y queda disponible para asignarle usuarios.
  4. Los usuarios autorizados comienzan a operar; todos los datos que generan nacen con el aplicacion_id de esa aplicación.
  5. Una aplicación puede activarse o desactivarse sin borrar sus datos.

Qué se clona y qué no al crear una aplicación nueva

✔ Sí se clona (estructura)

  • Categorías base
  • Estados de proyecto base
  • Configuraciones generales base
  • Parámetros funcionales mínimos
  • Estructura de permisos

✗ No se copia (datos de negocio)

  • Personas / desarrolladores
  • Requerimientos, entregas, facturación
  • Asignaciones, proyectos, sprints, roadmap
  • Tarifas, estimaciones, capacidades, horas
  • Historial de bitácora / auditoría
  • Datos de Azure DevOps y work items
  • Información importada de Excel

En una frase: se clona la estructura, no la data del negocio.

Resolución de la aplicación activa

  1. Tras el login, el frontend muestra un selector de aplicación con las aplicaciones autorizadas para el usuario. Para los roles superadmin y admin_app el selector incluye además la opción "Todas las aplicaciones" (vista consolidada).
  2. La aplicación elegida viaja en cada petición mediante una cabecera X-Aplicacion (y/o como reclamo en el token).
  3. El middleware de resolución valida que el usuario tenga acceso a esa aplicación e inyecta el aplicacion_id en el contexto de la petición.
  4. Todos los repositorios añaden aplicacion_id al filtro de forma obligatoria.
  5. Cambiar de aplicación en el frontend recarga todo el contexto.
Regla crítica de aislamiento Si una sola consulta o inserción no maneja el aplicacion_id, se mezclan datos entre aplicaciones. Mitigación: el filtro se centraliza en una capa base de repositorios (no se escribe a mano en cada consulta) y se cubre con pruebas de aislamiento.

Vista consolidada para administradores

El aislamiento estricto rige la operación diaria. Pero los roles de administración necesitan una mirada transversal — por eso el selector de aplicación ofrece el modo "Todas las aplicaciones", disponible solo para superadmin y admin_app.

AspectoModo operativo (una aplicación) Modo consolidado (todas)
QuiénTodos los roles Solo superadmin (todas) y admin_app (sus aplicaciones asignadas)

El aislamiento se mantiene El modo consolidado no "mezcla" datos: los agrega usando la aplicación como una dimensión más — cada cifra sigue identificando a qué aplicación pertenece — y solo para lectura. Cualquier creación o edición exige seleccionar una aplicación concreta; el middleware de resolución rechaza las escrituras hechas en modo consolidado.

El Dashboard unificado es la pantalla que aprovecha este modo: presenta gráficas consolidadas (requerimientos por estado, cumplimiento de ANS, horas estimadas, facturación y carga del equipo) apiladas o comparadas por aplicación, de modo que un administrador ve el panorama completo de todas sus aplicaciones en una sola vista.

6Autenticación y autorización

Quién entra, cómo se identifica y qué puede hacer.

Autenticación

  • Inicio de sesión con usuario y contraseña; las contraseñas se almacenan con bcrypt.
  • La sesión se mantiene con un token JWT firmado por el backend.
  • Endpoints de sesión: POST /api/auth/login, GET /api/auth/me, cierre de sesión en el cliente (descarte del token).
  • El token transporta: identificador de usuario, rol y aplicaciones autorizadas.

Roles de plataforma

La nueva versión define cuatro roles. Se guardan en la colección usuarios y controlan el acceso al sistema y al módulo de administración.

RolPuedeVisibilidad de datos
superadmin Crear, editar, activar/desactivar cualquier aplicación; asignar usuarios; entrar al módulo de administración. Todas las aplicaciones. Accede a la vista consolidada y al dashboard unificado de toda la plataforma.
admin_app Administrar las aplicaciones asignadas y su configuración interna. Sus aplicaciones asignadas. Vista consolidada y dashboard unificado limitados a ese conjunto.
editor Operar la información de su aplicación; no puede crear aplicaciones. Solo la aplicación activa.
viewer Solo consultar; no modifica ni crea. Solo la aplicación activa.
Los roles admin ven todo superadmin y admin_app no quedan atrapados en una sola aplicación: pueden cambiar al modo consolidado y obtener un panorama completo de todas sus aplicaciones autorizadas en el dashboard unificado (ver sección 5).
Dos niveles de rol — no se confunden El rol de plataforma (superadmin / admin_app / editor / viewer) vive en usuarios y define qué puede hacer en el sistema. El rol operativo (LT HITSS, LT EPM, EPM, COORD, DEV…) vive en personas y describe la función de esa persona en el dominio. Una persona operativa que además inicia sesión se vincula con su cuenta de usuario por email.

Autorización en cada petición

  1. El middleware de sesión valida el JWT y carga el usuario.
  2. El middleware de permisos verifica el rol/permiso requerido por el endpoint.
  3. El middleware de resolución de aplicación valida que el usuario tenga acceso a la aplicación activa.
  4. El endpoint se ejecuta con el aplicacion_id ya fijado en el contexto.

7Modelo de datos MongoDB unificado

Las colecciones de la BD hitss_unificado y su criterio de embebido vs. referencia.

Criterio de modelado Se embebe lo que se lee siempre junto y es acotado (la solicitud y las entregas dentro del requerimiento; los sprints dentro del proyecto). Se referencia por ObjectId lo que se consulta de forma independiente o crece sin límite (personas, bitácora, work items). Toda colección operativa lleva además aplicacion_id.

10Migración de datos existentes

Cómo se trasladan los datos de los dos SQLite a MongoDB bajo la aplicación inicial.

Se desarrolla scripts/migrar_a_mongo.py, que abre ambos archivos SQLite en modo solo lectura y vuelca los datos a MongoDB en orden de dependencias. Todo registro migrado recibe aplicacion_id = aplicación inicial (p. ej. código epm-hitss).

Orden de carga

  1. Aplicación inicial: crear el documento en aplicaciones.
  2. Usuarios: crear el superadmin y, a partir de las Persona con contraseña del Liquidador, las cuentas de usuarios.
  3. Catálogos base: squads, aplicativos (con dirección embebida), tarifas, festivos, categorias, configuracion.
  4. Personas: insertar las Persona del Liquidador; cruzar cada developer de WM por email — si coincide, fusionar campos de equipo; si no, crear persona con rol DEV.
  5. Requerimientos: insertar desde el Liquidador (con solicitud, entregas y facturación embebidas); cruzar el project_catalog de WM por req_code/code para completar categoría, developers y fechas; los sin coincidencia se crean marcados para revisión.
  6. Asignaciones de carga: migrar asignaciones → proyectos → sprints, reapuntando al _id del requerimiento.
  7. Resto: azdo_*, estimaciones, capacidades, bitacora, alertas.
  8. Reporte de discrepancias y validación de conteos origen vs. destino.

Migraciones de prueba + corte final

Siguiendo el manual de la nueva versión: la aplicación origen sigue cambiando mientras se desarrolla. Por eso se hacen una o varias migraciones de prueba en un entorno de pruebas y una migración final de corte que toma el estado más reciente, conservando altas, bajas y cambios hechos hasta el momento del paso a producción.

Validaciones mínimas

  • Los conteos coinciden entre origen y destino.
  • Las relaciones (referencias por ObjectId) siguen funcionando.
  • El frontend muestra la información esperada y los filtros operan.
  • La aplicación inicial conserva toda su información histórica.
  • Un dato de la aplicación inicial no aparece en otra aplicación.
  • Una aplicación nueva aparece sin datos operativos.
Riesgo de la fusión El cruce de personas depende del email y el de requerimientos de los códigos. El script debe emitir un reporte de discrepancias (personas sin email, requerimientos sin par) para revisión manual antes del corte final.

12Riesgos y mitigaciones

Lo que puede salir mal y cómo se controla.

RiesgoImpactoMitigación
Mezcla de datos entre aplicaciones por falta de filtro aplicacion_id Crítico Centralizar el filtro en un repositorio base; pruebas de aislamiento; revisión de todo endpoint que reciba o devuelva datos operativos.
Migración final desactualizada (copia vieja)Alto Migraciones de prueba + corte final con la data más reciente; congelar el origen al ejecutar el corte.
Datos sin cruce en la fusión (personas / requerimientos)Alto Reporte de discrepancias del script; revisión manual previa al corte.
Pérdida de integridad referencial al no haber foreign keysAlto Validar referencias en la capa de servicios; índices únicos; embebido cuando la relación es 1:1 o 1:pocos.
Aplicación nueva mal inicializada (sin catálogos base)Medio Servicio de provisión obligatorio; pruebas que verifican la estructura base.
Usuarios sin permiso creando o administrando aplicacionesMedio Autorización por rol/permiso en cada endpoint; guard de rutas en el frontend.
Regresiones al rehacer la UI del Liquidador en ReactMedio Pruebas comparativas de cifras; mantener el sistema viejo en solo lectura durante el rollback.
Reescritura del backend Node a PythonMedio Portar router por router con su prueba; el comportamiento queda fijado por la API REST.
Respaldo manual de MongoDB localMedio mongodump programado; verificación periódica de los respaldos.
